# systematic-debugging

## 概述

随机修复浪费时间并制造新 bug。快速补丁掩盖潜在问题。

**核心原则：** 始终在尝试修复之前找到根本原因。修复症状是失败。

**违反这个过程的字面就是违反调试的精神。**

## 铁律

```
没有首先调查根本原因就不能修复
```

如果你没有完成第 1 阶段，你就不能提议修复。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产中的 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别使用当：**
- 在时间压力下（紧急情况使猜测变得诱人）
- "就一个快速修复"看起来显而易见
- 你已经尝试了多个修复
- 之前的修复没有工作
- 你不完全理解问题

**不要跳过当：**
- 问题看起来简单（简单 bug 也有根本原因）
- 你在赶时间（匆忙保证返工）
- 经理想要现在就修复（系统化比乱搞更快）

## 四个阶段

你必须在继续到下一个阶段之前完成每个阶段。

### 阶段 1：根本原因调查

**在尝试任何修复之前：**

1. **仔细阅读错误消息**
   - 不要跳过错误或警告
   - 它们通常包含确切的解决方案
   - 完全阅读堆栈跟踪
   - 记录行号、文件路径、错误代码

2. **一致地重现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果不能重现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么更改可能导致这个？
   - Git diff、最近的提交
   - 新依赖、配置更改
   - 环境差异

4. **在多组件系统中收集证据**

   当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库），在每个组件边界添加诊断：记录进出数据、验证环境/配置传播、检查每层状态。运行一次收集证据，分析哪里坏掉，再调查该组件。

5. **追踪数据流**

   **当错误在调用栈深处时：**

   有关完整的向后追踪技术，请参阅此目录中的 `root-cause-tracing.md`。

   **快速版本：**
   - 坏值从哪里起源？
   - 什么用坏值调用了这个？
   - 继续追踪直到找到源头
   - 在源头修复，不是症状

### 阶段 2：模式分析

**在修复之前找到模式：**

1. **找到工作示例**
   - 在同一代码库中找到类似的 рабочий 代码
   - 什么类似的在工作什么在坏？

2. **与参考比较**
   - 如果正在实现模式，完全阅读参考实现
   - 不要略读——阅读每一行
   - 在应用之前完全理解模式

3. **识别差异**
   - 工作和坏的是什么不同？
   - 列出每个差异，无论多小
   - 不要假设"那不重要"

4. **理解依赖**
   - 这需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

### 阶段 3：假设和测试

**科学方法：**

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根本原因因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **最小化测试**
   - 做最小的可能更改来测试假设
   - 一次一个变量
   - 不要一次修复多个东西

3. **在继续之前验证**
   - 有效？→ 阶段 4
   - 无效？→ 形成新假设
   - 不要在上面添加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 进一步研究

### 阶段 4：实施

**修复根本原因，不是症状：**

1. **创建失败的测试用例**
   - 最简单的可能重现
   - 如果可能自动化测试
   - 如果没有框架一次性测试脚本
   - 修复前必须有
   - 使用 `supercoder:test-driven-development` skill 编写适当的失败测试

2. **实施单一修复**
   - 解决 identified 的根本原因
   - 一次一个更改
   - 没有"我在此处"的改进
   - 没有捆绑的重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没坏？
   - 问题实际解决了？

4. **如果修复不工作**
   - 停止
   - 计数：你尝试了多少个修复？
   - 如果 < 3：返回阶段 1，用新信息重新分析
   - **如果 ≥ 3：停止并质疑架构（见下面的步骤 5）**
   - **没有架构讨论不要尝试修复 #4**

5. **如果 3+ 个修复失败：质疑架构**

   **表示架构问题的模式：**
   - 每个修复揭示在不同位置的新共享状态/耦合/问题
   - 修复需要"大规模重构"才能实施
   - 每个修复在其他地方制造新症状

   **停止并质疑基础：**
   - 这个模式从根本上健全吗？
   - 我们是"纯粹因为惯性而坚持"？
   - 我们应该重构架构而不是继续修复症状？

   **在尝试更多修复之前与你的 human partner 讨论**

   这不是失败的假设——这是错误的架构。

## 红线与常见合理化

| 借口 | 现实 |
|------|------|
| "问题简单，不需要过程" | 简单问题也有根本原因。过程对简单 bug 很快。 |
| "紧急，没时间走过程" | systematic-debugging 比猜测乱搞**更快**。 |
| "先试试这个，然后调查" | 第一个修复设置模式。从一开始就做对。 |
| "一次多个修复节省时间" | 无法隔离什么工作。造成新 bug。 |
| "我看到问题了，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再尝试一次修复"（2+ 次失败） | 3+ 次失败 = 架构问题。质疑模式，不要再次修复。 |

**看到以上任何想法：停止。返回阶段 1。**
**3+ 个修复失败：质疑架构（见阶段 4.5）。**

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|------|----------|----------|
| **1. 根本原因** | 阅读错误、重现、检查更改、收集证据 | 理解什么和为什么 |
| **2. 模式** | 找到工作示例、比较 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决、测试通过 |

## 当调查揭示"无根本原因"

95% 的"无根本原因"是调查不完整。但如果确实是环境性/外部的：记录调查内容，实施处理（重试、超时、错误消息），添加监控。
