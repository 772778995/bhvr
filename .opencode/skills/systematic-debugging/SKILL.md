# systematic-debugging

## 概述

随机修复浪费时间并制造新 bug。快速补丁掩盖潜在问题。

**核心原则：** 始终在尝试修复之前找到根本原因。修复症状是失败。

**违反这个过程的字面就是违反调试的精神。**

## 铁律

```
没有首先调查根本原因就不能修复
```

如果你没有完成第 1 阶段，你就不能提议修复。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产中的 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别使用当：**
- 在时间压力下（紧急情况使猜测变得诱人）
- "就一个快速修复"看起来显而易见
- 你已经尝试了多个修复
- 之前的修复没有工作
- 你不完全理解问题

**不要跳过当：**
- 问题看起来简单（简单 bug 也有根本原因）
- 你在赶时间（匆忙保证返工）
- 经理想要现在就修复（系统化比乱搞更快）

## 四个阶段

你必须在继续到下一个阶段之前完成每个阶段。

### 阶段 1：根本原因调查

**不要自己读文件。派遣"调试侦察子代理"。**

主代理基于用户提供的信息，派遣一个调试侦察子代理，提供以下上下文给子代理：
- 用户描述的错误消息 / 测试失败输出
- 问题发生的位置（文件/函数名，如果已知）
- 相关背景（最近做了什么改动等）

**子代理负责执行以下调查：**

1. **仔细阅读错误消息**
   - 完整读取堆栈跟踪
   - 记录行号、文件路径、错误代码
   - 不要跳过任何警告

2. **检查最近的更改**
   - 运行 `git log --oneline -10`
   - 运行 `git diff HEAD~5..HEAD -- [相关文件]`
   - 检查新依赖、配置更改、环境差异

3. **在多组件系统中收集证据**

   当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库），在每个组件边界检查：进出数据、环境/配置传播、每层状态。

4. **追踪数据流**

   有关完整的向后追踪技术，请参阅此目录中的 `root-cause-tracing.md`。

   快速版本：坏值从哪里起源？什么用坏值调用了这个？继续追踪直到找到源头。

**子代理返回以下结构化报告（不是文件内容，只是引用）：**

```
根本原因候选：[按可能性排序的假设列表]
相关文件：[file:line 引用，不是文件内容]
最近变更：[可疑的变更列表]
证据摘要：[发现了什么、还缺什么]
```

### 阶段 2：模式分析

**基于阶段 1 子代理的报告，派遣模式分析子代理（可以是同一子代理的续集）。**

主代理把阶段 1 报告转发给子代理，附加以下指令：

1. **找到工作示例**
   - 在同一代码库中找到类似的工作代码
   - 什么类似的在工作，什么在坏？

2. **与参考比较**
   - 如果正在实现某个模式，完整阅读参考实现（每一行，不要略读）
   - 在应用之前完全理解该模式

3. **识别差异**
   - 列出工作代码和问题代码的每个差异，无论多小
   - 不要假设"那不重要"

4. **理解依赖**
   - 这需要什么其他组件？
   - 什么设置、配置、环境？
   - 它做什么假设？

**子代理返回以下结构化报告：**

```
工作示例位置：[file:line 引用]
关键差异：[工作代码 vs 问题代码的具体差异列表]
依赖/配置差异：[配置或环境差异]
```

### 阶段 3：假设和测试

**主代理基于阶段 1 + 2 子代理的结构化报告，做纯推理工作。主代理不读文件。**

1. **形成单一假设**
   - 清楚陈述："我认为 X 是根本原因因为 Y"
   - 写下来
   - 要具体，不要模糊

2. **用 `question` 工具确认方向**
   - 向 human partner 呈现假设和依据
   - 请求确认或纠正方向
   - 不要在未确认的假设上直接进入修复

3. **最小化测试策略**
   - 规划最小的可能更改来验证假设
   - 一次一个变量
   - 不要计划同时修复多个东西

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助或指示子代理进一步调查

### 阶段 4：实施

**使用 `supercoder:subagent-coordination` 派遣执行型子代理实施修复。**

主代理向执行子代理提供：
- 确认的根本原因假设（来自阶段 3）
- 需要修改的文件（来自阶段 1 子代理报告的 file:line 引用）
- 修复方向和约束

**执行子代理负责：**

1. **创建失败的测试用例**
   - 最简单的可能重现
   - 如果可能自动化测试
   - 如果没有框架一次性测试脚本
   - 修复前必须有
   - 遵循 `supercoder:test-driven-development` skill

2. **实施单一修复**
   - 解决 identified 的根本原因
   - 一次一个更改
   - 没有"顺手改进"
   - 没有捆绑的重构

3. **验证修复**
   - 测试现在通过了吗？
   - 其他测试没坏？
   - 问题实际解决了？

**执行子代理以标准格式返回：DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT**

4. **如果修复不工作**
   - 停止
   - 计数：你尝试了多少个修复？
   - 如果 < 3：主代理返回阶段 1，用新信息重新派遣侦察子代理
   - **如果 ≥ 3：停止并质疑架构（见下面的步骤 5）**
   - **没有架构讨论不要尝试修复 #4**

5. **如果 3+ 个修复失败：质疑架构**

   **表示架构问题的模式：**
   - 每个修复揭示在不同位置的新共享状态/耦合/问题
   - 修复需要"大规模重构"才能实施
   - 每个修复在其他地方制造新症状

   **停止并质疑基础：**
   - 这个模式从根本上健全吗？
   - 我们是"纯粹因为惯性而坚持"？
   - 我们应该重构架构而不是继续修复症状？

   **在尝试更多修复之前与你的 human partner 讨论**

   这不是失败的假设——这是错误的架构。

## 红线与常见合理化

| 借口 | 现实 |
|------|------|
| "问题简单，不需要过程" | 简单问题也有根本原因。过程对简单 bug 很快。 |
| "紧急，没时间走过程" | systematic-debugging 比猜测乱搞**更快**。 |
| "先试试这个，然后调查" | 第一个修复设置模式。从一开始就做对。 |
| "一次多个修复节省时间" | 无法隔离什么工作。造成新 bug。 |
| "我看到问题了，让我修复它" | 看到症状 ≠ 理解根本原因。 |
| "再尝试一次修复"（2+ 次失败） | 3+ 次失败 = 架构问题。质疑模式，不要再次修复。 |
| "我自己读一下文件更快" | 污染主代理上下文，降低后续推理质量。派子代理。 |
| "不需要派子代理，我直接跑命令" | 阶段 1-2 的文件读取和命令执行属于侦察子代理，不属于主代理。 |

**看到以上任何想法：停止。返回阶段 1。**
**3+ 个修复失败：质疑架构（见阶段 4.5）。**

## 快速参考

| 阶段 | 执行者 | 关键活动 | 输出 |
|------|--------|----------|------|
| **1. 根本原因** | 侦察子代理 | 读错误、跑 git、收集证据 | 结构化根本原因报告 |
| **2. 模式** | 模式分析子代理 | 找工作示例、比较差异 | 结构化差异报告 |
| **3. 假设** | 主代理（纯推理） | 基于报告形成假设、用 question 确认 | 确认的假设 |
| **4. 实施** | 执行子代理 | TDD 修复、验证 | DONE/BLOCKED/NEEDS_CONTEXT |

## 当调查揭示"无根本原因"

95% 的"无根本原因"是调查不完整。但如果确实是环境性/外部的：记录调查内容，实施处理（重试、超时、错误消息），添加监控。
