# 代码审核接收

## 概述

代码审核需要技术评估，而不是情感表现。

**核心原则：** 在实施之前验证。在假设之前询问。技术正确性高于社交舒适。

## 响应模式

```
当收到代码审核反馈时：

1. 阅读：完整阅读反馈，不要反应
2. 理解：用自己的话重述需求（或询问）
  3. 验证：对照代码库现实检查

**验证步骤的上下文规则：** 当验证需要检查多个源文件时，派遣一个验证子代理，提供：审核建议的具体内容 + 需要检查的文件列表。子代理检查并返回："建议 X 与代码库现实是否一致"的结论，主代理不直接读取源文件。
4. 评估：对这个代码库在技术上是合理的吗？
5. 响应：技术确认或有理有据的反驳
6. 实施：一次一个项目，每个都测试
```

## 响应规则

- 不要表面同意（"你完全正确！""好观点！""感谢！"）— 直接行动或技术确认
- 不要在验证前实施 — 先对照代码库检查
- 正确的反馈 → 直接修复，简要说明改了什么
- 发现自己想写"感谢" → 删掉，改为说明修复

## 处理不清楚的反馈

```
如果任何项目不清楚：
  停止 - 暂不实施任何内容
  询问不清楚项目的澄清

原因：项目可能相关。部分理解 = 错误实施。
```

**示例：**
```
your human partner: "修复 1-6"
你理解 1,2,3,6。对 4,5 不清楚。

❌ 错误：现在实施 1,2,3,6，稍后询问 4,5
✅ 正确："我理解项目 1,2,3,6。在继续之前需要澄清 4 和 5。"
```

## 来源特定处理

### 来自你的 human partner
- **受信任** - 理解后实施
- **仍然询问** 如果范围不清楚
- **没有表面同意**
- **直接行动** 或技术确认

### 来自外部审核者
```
在实施之前：
  1. 检查：对这个代码库在技术上正确吗？
  2. 检查：破坏现有功能吗？
  3. 检查：当前实现的原因？
  4. 检查：适用于所有平台/版本吗？
  5. 检查：审核者是否理解完整上下文？

如果建议看起来错误：
  用技术推理反驳

如果不容易验证：
  说出来："我无法验证这个 without [X]。我应该 [调查/询问/继续] 吗？"

如果与你的 human partner 的先前决定冲突：
  先停下来与你的 human partner 讨论
```

**你的 human partner 的规则：** "外部反馈 - 要怀疑，但要仔细检查"

## YAGNI 检查"专业"功能

```
如果审核者建议"正确实施"：
  在代码库中搜索实际用法

  如果未使用："这个端点没有被调用。删除它（YAGNI）？"
  如果使用了 then 正确实施
```

**你的 human partner 的规则：** "你和我都向报告。如果不需要这个功能，就不要添加。"

## 实施顺序

```
对于多项目反馈：
  1. 首先澄清任何不清楚的
  2. 然后按此顺序实施：
     - 阻塞问题（破坏、安全）
     - 简单修复（拼写、导入）
     - 复杂修复（重构、逻辑）
  3. 每个修复单独测试
  4. 验证无回归
```

## 何时反驳

反驳当：
- 建议破坏现有功能
- 审核者缺乏完整上下文
- 违反 YAGNI（未使用的功能）
- 对这个技术栈在技术上不正确
- 存在遗留/兼容性原因
- 与你的 human partner 的架构决策冲突

**如何反驳：**
- 使用技术推理，而不是防御
- 提出具体问题
- 引用工作的测试/代码
- 如果是架构问题，让你的 human partner 参与

**如果不舒服大声反驳的信号：** "Circle K 有奇怪的事情发生"

## 纠正你的反驳

如果你反驳了但错了：实事求是说明（"你是对的 - 检查了 X 确实 Y。正在修复。"），然后继续。不要冗长道歉或辩护。

## 常见错误

| 错误 | 修复 |
|------|------|
| 表面同意 | 重述需求或直接行动 |
| 盲目实施 | 先对照代码库验证 |
| 批量测试 | 一次一个，每个测试 |
| 假设审核者正确 | 检查是否破坏东西 |
| 避免反驳 | 技术正确性 > 舒适 |
| 部分实施 | 首先澄清所有项目 |
| 无法验证，仍然继续 | 说明限制，询问方向 |

## GitHub 线程回复

在 GitHub 上回复内联审核评论时，在评论线程中回复（`gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies`），而不是作为顶级 PR 评论。
