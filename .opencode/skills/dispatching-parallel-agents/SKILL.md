# 调度并行 Agents

## 概述

这是一个轻量参考 skill，只解决一件事：当多个独立任务可以并发推进时，如何把它们安全地拆给多个 agent。

**核心原则：** 独立任务才并行；边界不清就不要并行。

**与 `subagent-driven-development` 的边界：**
- 如果你在执行规格中的实现任务，使用 `supercoder:subagent-driven-development`（历史仓库里的旧计划文档也归它处理）
- 只有当任务不需要它的依赖分析、两阶段审核和实现控制器流程时，才使用本 skill

## 何时使用

**使用当：**
- 2+ 个任务彼此独立
- 任务之间没有共享文件、共享状态或顺序依赖
- 你只是想并行调查、检索、审阅或做小范围修改
- 不需要规格合规审核或实现批次控制

**不使用当：**
- 你在执行一个规格
- 任务之间会改同一文件
- 你不确定它们是否独立
- 你需要统一的实现后审核和修复循环

## 快速检查

在并行调度前，逐项确认：

- 每个任务都能用一句话单独定义
- 每个任务需要的上下文可以单独给全
- 每个任务的输出彼此独立
- 一个任务失败不会让另一个任务的结论失效
- 不存在共享文件编辑风险

有任一项不确定，就顺序执行。

## 模式

### 1. 按问题域拆分

好例子：
- 调查 `test/a.test.ts` 失败
- 调查 `test/b.test.ts` 失败
- 审阅 `docs/README.md` 的术语问题

坏例子：
- 三个 agent 同时修改同一模块
- 上游接口未定时并行做依赖它的改动

### 2. 给每个 agent 自包含上下文

每个 agent 至少要有：
- 明确范围
- 明确目标
- 明确禁止项
- 明确返回格式

示例：

```markdown
检查 `src/foo.test.ts` 的失败原因。

目标：判断这是测试脆弱性还是生产代码 bug。
约束：不要修改其他文件。
返回：根因、建议修复点、是否与其他失败相关。
```

### 3. 汇总后统一验证

所有 agent 返回后：
- 先看是否冲突
- 再决定是否合并结果
- 最后运行相关验证命令

## 常见错误

**把相关任务硬拆开**
- 问题：多个 agent 基于不同假设同时推进，结果互相打架
- 修复：先顺序澄清共享前提

**prompt 不自包含**
- 问题：agent 需要回头猜上下文，产出不稳定
- 修复：直接给文件、错误、目标和限制

**把实现工作流错放到这里**
- 问题：缺少依赖分析、审核和返工闭环
- 修复：改用 `supercoder:subagent-driven-development`

## 底线

这是一个并行调度参考 skill，不是实现总控 skill。

需要计划执行、依赖分析、两阶段审核时，用 `subagent-driven-development`。
