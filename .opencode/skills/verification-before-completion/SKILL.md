# verification-before-completion

## 概述

在没有验证的情况下声称工作完成是不诚实，不是效率。

**核心原则：** 证据永远在声明之前。

**违反这条规则的字面就是违反规则的精神。**

## 铁律

```
没有新鲜验证证据就不能声称完成
```

如果你没有在这条消息中运行验证命令，你就不能声称它通过了。

## 门控函数

```
在声称任何状态或表达满足之前：

1. 识别：什么命令能证明这个声明？
2. 运行：执行完整命令（新鲜的、完整的）
3. 读取：完整输出，检查退出代码，统计失败数
4. 验证：输出是否确认声明？
   - 如果否：用证据说明实际状态
   - 如果是：用证据声明
5. 只有这样：做出声明

跳过任何步骤 = 撒谎，不是验证
```

## 常见失败

| 声明 | 需要 | 不充分 |
|-------|----------|----------------|
| 测试通过 | 测试命令输出：0 失败 | 上次运行、"应该通过" |
| Linter 干净 | Linter 输出：0 错误 | 部分检查、推断 |
| 构建成功 | 构建命令：exit 0 | Linter 通过、日志看起来好 |
| Bug 修复 | 测试原始症状：通过 | 代码更改、假设修复 |
| 回归测试工作 | 红-绿循环验证 | 测试通过一次 |
| Agent 完成 | VCS diff 显示更改 | Agent 报告"成功" |
| 需求满足 | 逐行检查清单 | 测试通过 |

## 危险信号与合理化

| 信号 | 现实 |
|------|------|
| 使用"应该"、"可能"、"似乎" | 运行验证 |
| 验证前表达满足（"太棒了！""完成！"） | 证据在声明之前 |
| 即将 commit/push/PR 而不验证 | 运行命令 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 通过了" | Linter ≠ 编译器 |
| "Agent 说成功" | 独立验证 |
| "部分检查就够了" | 部分证明不了什么 |

## 何时应用

任何成功/完成声明之前、commit/PR/任务完成之前、委托给 agents 之后。没有捷径。
