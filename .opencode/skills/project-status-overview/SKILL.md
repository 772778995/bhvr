# project-status-overview

## 概述

用最低成本建立当前项目的进展感知。

**核心原则：** 先看文档状态，再看工作区状态，最后给出当前焦点与下一步。

**开始时宣布：** "我正在使用 project-status-overview skill 来快速获取当前项目进展。"

## 何时使用

- 开始一个新任务前，不确定当前已有功能推进到哪里
- 发现仓库里有多个功能文档，需要快速判断哪个是当前主线
- 接手中断过的工作，需要先看是否有 `进行中` 或 `已搁置` 的文档
- 想确认当前工作区改动是否和现有功能文档一致

## 快速流程

1. 查看 `docs/` 目录结构，找到功能文档（`docs/<类别>/<功能名>/README.md`）
2. 优先识别状态字段为 `进行中` 的文档
3. 如果没有 `进行中`，按优先级依次查看：最近修改的 `已搁置`、最近修改的 `已完成`
4. 打开最相关的 1-3 个文档，读取标题、状态、设计结论、变更历史和实施任务
5. 结合 `git status` 与最近提交，判断代码改动是否正在实现这些文档中的内容
6. 输出简短中文概览：当前焦点、活动文档、阻塞项、下一步

## 优先级规则

按这个顺序判断当前项目焦点：

1. 状态为 `进行中` 的功能文档
2. 最近修改的 `已搁置` 文档
3. 最近修改的 `已完成` 文档（用于理解刚完成了什么）

功能文档可能包含实施任务部分（带 checkbox 追踪）。

## 输出格式

输出保持简短，包含这 4 项：

```markdown
**当前焦点**
[一句话说明当前最可能正在推进的工作]

**相关文档**
- `[状态]` `docs/<类别>/<功能名>/README.md`

**当前判断**
- [代码改动是否与文档一致]
- [是否存在阻塞或状态不一致]

**下一步**
1. [最合理的下一步动作]
```

## 检查点

- 不要只看文件名；至少读最相关文档里的状态字段和设计结论
- 不要只看文档；要结合 `git status` 或最近提交判断是不是还在推进
- 如果没有任何相关文档，明确说"当前没有可用的功能文档"

## 常见错误

**只按最新修改时间判断主线**
- **问题：** 可能把刚修订过的旧文档误判为当前焦点
- **修复：** 先看状态字段，再结合修改时间

**只看文档，不看 git 状态**
- **问题：** 无法知道当前代码改动是否已经偏离文档
- **修复：** 用 `git status` 和最近提交做交叉验证

## 红线

- 不要把历史 `已完成` 文档直接当成当前焦点，除非没有更活跃的文档
- 不要在没读正文的情况下，仅凭目录名下结论

## 集成

**开始前可搭配：**
- `brainstorming`：先获取当前项目状态，再判断新需求是否与现有方向冲突

**完成后可转入：**
- 当前焦点已明确：进入对应 skill 执行
- 当前状态混乱：先修正文档状态，再继续开发
