# project-status-overview

## 概述

用最低成本建立当前项目的进展感知。

**核心原则：** 派遣 `@status-scout` 子代理完成所有文件读取和 git 操作，主代理只消费结构化摘要，保持上下文干净。

**开始时宣布：** "我正在使用 project-status-overview skill 来快速获取当前项目进展。"

## 何时使用

- 开始一个新任务前，不确定当前已有功能推进到哪里
- 发现仓库里有多个功能文档，需要快速判断哪个是当前主线
- 接手中断过的工作，需要先看是否有 `进行中` 或 `已搁置` 的文档
- 想确认当前工作区改动是否和现有功能文档一致

## 执行方式：派遣子代理

**主代理不直接读文件，不直接运行 git 命令。**

派遣 `@status-scout` 子代理，无需额外传入指令（代理已内置扫描逻辑和输出格式）。

等待 `@status-scout` 返回摘要后，主代理基于摘要做判断和决策，不再重新读取原始文件。

**如果 docs/ 目录不存在或没有功能文档：** status-scout 应明确返回"当前没有可用的功能文档"，主代理直接告知用户，不需要继续探索。

## 优先级规则（给 status-scout 参考）

按这个顺序判断当前项目焦点：

1. 状态为 `进行中` 的功能文档
2. 最近修改的 `已搁置` 文档
3. 最近修改的 `已完成` 文档（用于理解刚完成了什么）

功能文档可能包含实施任务部分（带 checkbox 追踪）。

## 主代理使用摘要的方式

收到摘要后：
- **直接基于摘要做判断**，不重新读原始文件
- **只有当摘要信息不足以做判断时**，才针对性地请求子代理补充特定字段
- **不要把摘要当成开始点再全量读文件**——摘要已经是结论，不是线索

## 红线

**主代理禁止直接执行的操作：**
- 用 Read tool 打开 docs/ 下的文件
- 直接运行 `git status` 或 `git log`
- 在收到摘要前就开始判断

**这些想法意味着你在绕过子代理模式——停下：**

| 想法 | 现实 |
|------|------|
| "只是快速看一下文件，派子代理太慢了" | 慢？派遣子代理是单条消息。读文件会把几百行内容压进你的上下文。 |
| "子代理也是我，上下文是共享的" | 子代理在独立会话中运行，不污染主代理上下文 |
| "只有一个文档，我直接读就好" | 就算只有一个文档，规则也一样。建立习惯。 |
| "git status 只是一条命令，无所谓" | 今天 git status，明天 git log，后天 git diff HEAD~10。主代理上下文是稀缺资源。 |

## 常见错误

**只按最新修改时间判断主线**
- **问题：** 可能把刚修订过的旧文档误判为当前焦点
- **修复：** 先看状态字段，再结合修改时间（交给 status-scout 判断）

**收到摘要后再读一遍原文档"验证"**
- **问题：** 摧毁了子代理模式的意义
- **修复：** 信任摘要。有疑问时让 status-scout 补充具体字段，而不是自己重读

## 集成

**开始前可搭配：**
- `brainstorming`：先获取当前项目状态，再判断新需求是否与现有方向冲突

**完成后可转入：**
- 当前焦点已明确：进入对应 skill 执行
- 当前状态混乱：先修正文档状态，再继续开发
