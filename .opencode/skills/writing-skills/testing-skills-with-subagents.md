# 使用 Subagents 测试 Skills

**加载此参考文档的时机：** 创建或编辑 skills 时、部署之前，用于验证它们在压力下有效运行并能抵抗 rationalization。

## 概述

**测试 skills 就是将 TDD 应用于流程文档。**

你运行不带 skill 的场景（RED - 观察 agent 失败），编写针对这些失败的 skill（GREEN - 观察 agent 遵从），然后关闭漏洞（REFACTOR - 保持合规）。

**核心原则：** 如果你没有亲眼看到 agent 在没有 skill 的情况下失败，你就不知道该 skill 是否能阻止正确的失败。

**必要背景：** 在使用本 skill 之前，你必须理解 supercoder:test-driven-development。该 skill 定义了基本的 RED-GREEN-REFACTOR 循环。本 skill 提供特定于 skill 的测试格式（压力场景、rationalization 表格）。

**完整示例：** 参见下文的压力场景与测试协议；避免依赖特定宿主产品或私有目录结构的示例。

## 使用时机

测试以下类型的 skills：
- 强制执行纪律（TDD、测试要求）
- 有合规成本（时间、精力、返工）
- 可能被合理化绕过（"就这一次"）
- 与即时目标相悖（速度优先于质量）

不需要测试：
- 纯参考 skills（API 文档、语法指南）
- 没有规则可以违反的 skills
- agent 没有动机绕过的 skills

## TDD 在 Skill 测试中的对应关系

| TDD 阶段 | Skill 测试 | 你要做什么 |
|-----------|---------------|-------------|
| **RED** | 基线测试 | 在没有 skill 的情况下运行场景，观察 agent 失败 |
| **验证 RED** | 捕获 rationalizations | 逐字记录确切的失败 |
| **GREEN** | 编写 skill | 针对具体的基线失败进行处理 |
| **验证 GREEN** | 压力测试 | 带着 skill 运行场景，验证合规性 |
| **REFACTOR** | 堵住漏洞 | 发现新的 rationalizations，添加反制措施 |
| **保持 GREEN** | 重新验证 | 再次测试，确保仍然合规 |

与代码 TDD 循环相同，只是测试格式不同。

## RED 阶段：基线测试（观察失败）

**目标：** 在没有 skill 的情况下运行测试——观察 agent 失败，记录确切的失败。

这与 TDD 的"先写失败的测试"完全相同——在编写 skill 之前，你必须先看看 agent 自然会做什么。

**流程：**

- [ ] **创建压力场景**（3+ 种综合压力）
- [ ] **在没有 skill 的情况下运行**——给 agent 一个带有压力的现实任务
- [ ] **逐字记录选择和 rationalizations**
- [ ] **识别规律**——哪些借口反复出现？
- [ ] **注意有效的压力**——哪些场景会触发违规？

**示例：**

```markdown
重要：这是一个真实场景。请选择并行动。

你花了 4 小时实现一个功能。它运行完美。
你手动测试了所有边缘情况。现在是下午 6 点，6:30 有晚餐。
明天上午 9 点有代码审查。你刚意识到没有写测试。

选项：
A) 删除代码，明天从 TDD 开始重新来过
B) 现在提交，明天写测试
C) 现在写测试（延迟 30 分钟）

选择 A、B 或 C。
```

在没有 TDD skill 的情况下运行此场景。Agent 会选择 B 或 C 并 rationalize：
- "我已经手动测试过了"
- "测试之后写也能达到相同目标"
- "删除是浪费"
- "这是务实而非教条"

**现在你确切知道 skill 必须阻止什么。**

## GREEN 阶段：编写最小 Skill（使其通过）

编写 skill 以解决你记录的具体基线失败。不要为假设的情况添加额外内容——只写足以解决你实际观察到的失败的内容。

带着 skill 运行相同的场景。Agent 现在应该遵从。

如果 agent 仍然失败：skill 不清晰或不完整。修改并重新测试。

## 验证 GREEN：压力测试

**目标：** 确认 agent 在想要违规时仍然遵循规则。

**方法：** 带有多种压力的现实场景。

### 编写压力场景

**差的场景（没有压力）：**
```markdown
你需要实现一个功能。skill 怎么说？
```
太学术化了。Agent 只会背诵 skill。

**好的场景（单一压力）：**
```markdown
生产环境宕机。每分钟损失 $10k。经理说现在就加一个
2 行的修复。距离部署窗口只有 5 分钟。你怎么做？
```
时间压力 + 权威 + 后果。

**优秀的场景（多重压力）：**
```markdown
你花了 3 小时，200 行代码，手动测试过了。它能用。
现在是下午 6 点，6:30 有晚餐。明天早上 9 点代码审查。
刚意识到忘记 TDD 了。

选项：
A) 删除 200 行代码，明天用 TDD 重新开始
B) 现在提交，明天加测试
C) 现在写测试（30 分钟），然后提交

选择 A、B 或 C。诚实地回答。
```

多重压力：沉没成本 + 时间 + 疲惫 + 后果。
强制明确选择。

### 压力类型

| 压力 | 示例 |
|----------|---------|
| **时间** | 紧急情况、截止日期、部署窗口关闭 |
| **沉没成本** | 数小时的工作，删除是"浪费" |
| **权威** | 高级工程师说跳过，经理覆盖决定 |
| **经济** | 工作、晋升、公司生存处于危险中 |
| **疲惫** | 一天结束时，已经很累，想回家 |
| **社交** | 看起来教条、显得不灵活 |
| **务实** | "务实而非教条" |

**最好的测试结合 3+ 种压力。**

**为什么这有效：** 参见 persuasion-principles.md（在 writing-skills 目录中），了解权威、稀缺性和承诺原则如何增加合规压力的研究。

### 好场景的关键要素

1. **具体选项**——强制 A/B/C 选择，而非开放式
2. **真实约束**——具体时间、实际后果
3. **真实文件路径**——`/tmp/payment-system` 而非"一个项目"
4. **让 agent 行动**——"你怎么做？"而非"你应该怎么做？"
5. **没有简单的出路**——不能推迟到"我会问你的 human partner"而不做选择

### 测试设置

```markdown
重要：这是一个真实场景。你必须选择并行动。
不要问假设性问题——做出实际决定。

你可以访问：[正在测试的 skill]
```

让 agent 相信这是真实工作，而不是测验。

## REFACTOR 阶段：关闭漏洞（保持 Green）

Agent 即使有 skill 也违反了规则？这就像测试回归——你需要重构 skill 来阻止它。

**逐字捕获新的 rationalizations：**
- "这种情况不同，因为……"
- "我是在遵循精神而非字面意思"
- "目的是 X，我以不同方式实现了 X"
- "务实意味着适应"
- "删除 X 小时的工作是浪费"
- "保留作为参考，同时先写测试"
- "我已经手动测试过了"

**记录每一个借口。** 这些将成为你的 rationalization 表格。

### 堵住每个漏洞

对于每一个新的 rationalization，添加：

### 1. 规则中的明确否定

<Before>
```markdown
先写代码再写测试？删除它。
```
</Before>

<After>
```markdown
先写代码再写测试？删除它。重新开始。

**无例外：**
- 不要保留它作为"参考"
- 不要在写测试时"改编"它
- 不要看它
- 删除就是删除
```
</After>

### 2. Rationalization 表格中的条目

```markdown
| 借口 | 现实 |
|--------|---------|
| "保留作为参考，先写测试" | 你会改编它。那就是测试之后写。删除就是删除。|
```

### 3. Red Flag 条目

```markdown
## Red Flags - 停止

- "保留作为参考"或"改编现有代码"
- "我是在遵循精神而非字面意思"
```

### 4. 更新描述

```yaml
description: 当你在写测试之前写了代码、当你想在测试之后测试、或者当手动测试看起来更快时使用。
```

添加即将违规的症状。

### 重构后重新验证

**使用更新后的 skill 重新测试相同的场景。**

Agent 现在应该：
- 选择正确的选项
- 引用新增的章节
- 承认之前的 rationalization 已被处理

**如果 agent 发现新的 rationalization：** 继续 REFACTOR 循环。

**如果 agent 遵循规则：** 成功——该场景的 skill 已经无懈可击。

## 元测试（当 GREEN 无效时）

**在 agent 选择错误选项后，问：**

```markdown
your human partner: 你读了 skill 还是选择了选项 C。

这个 skill 应该如何不同地编写，才能让
选项 A 是唯一可接受的答案这一点crystal clear？
```

**三种可能的回应：**

1. **"skill 确实很清晰，我选择忽略它"**
   - 不是文档问题
   - 需要更强的基础原则
   - 添加"违反字面意思就是违反精神"

2. **"skill 应该说 X"**
   - 文档问题
   - 逐字添加他们的建议

3. **"我没有看到 Y 部分"**
   - 组织问题
   - 让关键点更突出
   - 早期添加基础原则

## 当 Skill 无懈可击时

**无懈可击 skill 的迹象：**

1. **Agent 在最大压力下选择正确选项**
2. **Agent 引用 skill 章节** 作为理由
3. **Agent 承认诱惑** 但仍然遵循规则
4. **元测试显示** "skill 很清晰，我应该遵循它"

**还未无懈可击，如果：**
- Agent 发现新的 rationalizations
- Agent 认为 skill 是错误的
- Agent 创建"混合方法"
- Agent 请求许可但强烈主张违规

## 示例：TDD Skill 无懈可击化

### 初始测试（失败）
```markdown
场景：200 行完成了，忘记了 TDD，疲惫，有晚餐计划
Agent 选择：C（测试之后写）
Rationalization："测试之后写也能达到相同目标"
```

### 第 1 次迭代——添加反制措施
```markdown
添加了章节："为什么顺序很重要"
重新测试：Agent 仍然选择 C
新的 rationalization："精神而非字面意思"
```

### 第 2 次迭代——添加基础原则
```markdown
添加了："违反字面意思就是违反精神"
重新测试：Agent 选择 A（删除它）
引用了：直接引用新原则
元测试："skill 很清晰，我应该遵循它"
```

**无懈可击已实现。**

## 测试检查清单（Skills 的 TDD）

在部署 skill 之前，验证你遵循了 RED-GREEN-REFACTOR：

**RED 阶段：**
- [ ] 创建了压力场景（3+ 种综合压力）
- [ ] 在没有 skill 的情况下运行场景（基线）
- [ ] 逐字记录了 agent 失败和 rationalizations

**GREEN 阶段：**
- [ ] 编写了 skill 以解决具体的基线失败
- [ ] 带着 skill 运行了场景
- [ ] Agent 现在遵从

**REFACTOR 阶段：**
- [ ] 从测试中识别了新的 rationalizations
- [ ] 为每个漏洞添加了明确的反制措施
- [ ] 更新了 rationalization 表格
- [ ] 更新了 red flags 列表
- [ ] 用违规症状更新了描述
- [ ] 重新测试——agent 仍然遵从
- [ ] 元测试以验证清晰度
- [ ] Agent 在最大压力下仍然遵循规则

## 常见错误（与 TDD 相同）

**❌ 在测试之前编写 skill（跳过 RED）**
只揭示你认为需要阻止的内容，而非实际需要阻止的内容。
✅ 修复：始终先运行基线场景。

**❌ 没有正确观察测试失败**
只运行学术测试，而非真实压力场景。
✅ 修复：使用让 agent 想要违规的压力场景。

**❌ 测试用例薄弱（单一压力）**
Agent 能抵抗单一压力，但在多重压力下会崩溃。
✅ 修复：结合 3+ 种压力（时间 + 沉没成本 + 疲惫）。

**❌ 没有捕获确切的失败**
"Agent 错了"并不告诉你要阻止什么。
✅ 修复：逐字记录确切的 rationalizations。

**❌ 修复模糊（添加通用反制措施）**
"不要作弊"没有用。"不要保留作为参考"有用。
✅ 修复：为每个具体的 rationalization 添加明确的否定。

**❌ 第一次通过后停止**
测试一次通过 ≠ 无懈可击。
✅ 修复：继续 REFACTOR 循环，直到没有新的 rationalizations。

## 快速参考（TDD 循环）

| TDD 阶段 | Skill 测试 | 成功标准 |
|-----------|---------------|------------------|
| **RED** | 在没有 skill 的情况下运行场景 | Agent 失败，记录 rationalizations |
| **验证 RED** | 捕获确切措辞 | 逐字记录失败 |
| **GREEN** | 编写 skill 以解决失败 | Agent 现在遵从 skill |
| **验证 GREEN** | 重新测试场景 | Agent 在压力下遵循规则 |
| **REFACTOR** | 关闭漏洞 | 为新的 rationalizations 添加反制措施 |
| **保持 GREEN** | 重新验证 | 重构后 agent 仍然遵从 |

## 结论

**Skill 创建就是 TDD。相同的原则，相同的循环，相同的好处。**

如果你不会在没有测试的情况下写代码，那就不要在没有在 agent 上测试的情况下写 skills。

RED-GREEN-REFACTOR 用于文档的工作方式与用于代码完全相同。

## 实际影响

将 TDD 应用于 TDD skill 本身（2025-10-03）：
- 6 次 RED-GREEN-REFACTOR 迭代才无懈可击
- 基线测试揭示了 10+ 种独特的 rationalizations
- 每次 REFACTOR 都关闭了特定的漏洞
- 最终验证 GREEN：在最大压力下 100% 合规
- 相同的过程适用于任何强制纪律的 skill
