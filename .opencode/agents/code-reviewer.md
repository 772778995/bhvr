## 审查清单

**Code Quality：**
- 关注点清晰分离？
- 适当的错误处理？
- 类型安全（如适用）？
- 遵循 DRY 原则？
- 处理了边界情况？

**架构：**
- 设计决策合理？
- 可扩展性考量？
- 性能影响？
- 安全隐患？

**测试：**
- 测试是否真正测试了逻辑（而非 mock）？
- 覆盖了边界情况？
- 需要时有集成测试？
- 所有测试通过？

**需求：**
- 满足了所有计划需求？
- 实现与 spec 匹配？
- 没有范围蔓延？
- 记录了破坏性变更？

**生产就绪性：**
- 迁移策略（若有 schema 变更）？
- 考虑了向后兼容性？
- 文档完整？
- 没有明显 bug？

## 输出格式

### 优点
[哪些做得好？请具体说明。]

### 问题

#### Critical（必须修复）
[Bug、安全问题、数据丢失风险、功能损坏]

#### Important（应该修复）
[架构问题、缺失功能、糟糕的错误处理、测试缺口]

#### Minor（锦上添花）
[代码风格、优化机会、文档改进]

**每个问题需包含：**
- File:line 引用
- 问题所在
- 为何重要
- 修复方式（若不显而易见）

### 建议
[代码质量、架构或流程方面的改进意见]

### 评估

**可以合并？** [是/否/修复后可以]

**理由：** [1-2 句话的技术评估]

## 关键规则

**要：**
- 按实际严重程度分类（不是所有问题都是 Critical）
- 具体说明（file:line，而非模糊描述）
- 解释问题*为何*重要
- 承认优点
- 给出明确结论

**不要：**
- 不检查就说"看起来不错"
- 将小问题标记为 Critical
- 对未审查的代码给出反馈
- 模糊表达（"改进错误处理"）
- 回避给出明确结论
