## 设计结论

`快速找书` 从主链移除 `Google Books API`，保留 `Open Library` 作为公开主题检索源，并增加基于 `ISBN` 的微信读书元数据桥接：通过第三方 metadata API 补齐 `weread_url`、微信读书简介与评分映射；桥接失败或缺字段时，结果必须明确显示 `暂无数据`，而不是回退到 Google 或伪造字段。

## 背景判断

- 产品已明确要求快速找书不能使用 Google 接口。
- 现有实现主链仍直接调用 `https://www.googleapis.com/books/v1/volumes`，与产品要求冲突。
- 微信读书没有本项目已验证的官方公开结构化搜索 API，直接依赖民间逆向私接口稳定性和合规性都很差。
- 当前可落地的折中方案，是保留公开主题检索源负责找候选，再通过第三方 metadata API 做微信读书链接桥接。
- 这不是“腾讯官方 API 接入”，而是产品指定方向下相对可维护、影响面最小的实现。

## 范围边界

本次实现：

- 删除快速找书主链中的 `Google Books API`
- 保留 `Open Library` 负责主题检索和候选书目归一化
- 按候选书的 `ISBN` 调用第三方 metadata API，补充微信读书链接与相关元数据
- 渲染结果时，把微信读书可读性/评分/入口改成“有数据就展示，没有就写 `暂无数据`”
- 新增环境变量占位，用于 metadata API 的地址与鉴权
- 补充测试，锁定“不会再访问 Google”与“桥接失败时显示暂无数据”

本次不实现：

- 不接入 Google 作为任何形式的兜底源
- 不自行抓取微信读书网页或私有 RPC
- 不承诺所有结果都一定有微信读书入口
- 不伪造微信读书评分、是否可读或链接

## 推荐架构

### 1. 候选检索

- 继续使用 `Open Library Search API` 做主题检索
- 只保留带 `isbn` 的候选作为微信读书桥接优先项
- 若缺少 `isbn`，仍可作为普通书目候选展示，但微信读书字段显示 `暂无数据`

### 2. 微信读书桥接

- 新增 metadata API 配置：
  - `BOOK_METADATA_BASE_URL`
  - `BOOK_METADATA_API_KEY`
- 对每本带 `isbn` 的候选调用 `GET /metadata?isbn=...`
- 从返回中读取：
  - `weread_url`
  - `weread_intro`
  - `douban_rating`（仅当作公开评分来源，不冒充微信读书评分）
- metadata API 返回错误、超时、401/403/404 或缺字段时，统一视为桥接失败

### 3. 输出规则

- 结果文案继续声明：书目信息来自公开数据源与第三方桥接，不由模型补写
- 书目卡中的“线上平台与评分”规则改为：
  - 有公开评分则显示其来源与数值
  - 微信读书链接存在则显示链接
  - 微信读书链接不存在或桥接失败则显示 `微信读书：暂无数据`
- 不再出现 `Google Books` 文案

### 4. 失败退化

- `Open Library` 成功、metadata API 失败：仍返回书目，但微信读书字段统一 `暂无数据`
- `Open Library` 无结果：返回当前已有的“暂无可靠结果”空态
- metadata API 不得阻断整次找书请求

## 验证方式与成功标准

- 服务端快速找书链路不再请求 `Google Books API`
- 查询结果里不再出现 `Google Books` 文案
- 微信读书桥接成功时，结果包含微信读书入口
- 微信读书桥接失败时，结果明确显示 `微信读书：暂无数据`
- 新增环境变量缺失时有明确错误或安全降级，不会静默乱跑

## 自审结果

- 已明确这轮实现不是官方微信读书 API，而是第三方 metadata 桥接
- 已明确产品要求是“禁用 Google”，不是“必须保证微信读书始终有数据”
- 已明确桥接失败不能阻断主流程，只能退化为 `暂无数据`
- 已明确这是单任务规格，无需拆总览文档
