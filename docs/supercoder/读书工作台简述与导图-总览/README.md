## 设计结论

`/book` 右栏当前的短版阅读入口不该继续挂着“快速读书”这种半成品名字，而应明确改成更贴近产出形态的`书籍简述`；对应内置 preset 也同步改成 300 字内的短版提示。`详细解读` 则收口到用户给定的 5000 字内结构化长版提示。思维导图功能不建议直接照着“让 NotebookLM 吐一段 Mermaid，前端像 Typora 一样渲染”就草率上线；推荐方向仍是让 NotebookLM 优先输出 Mermaid `mindmap` 代码块，但前端必须补生成约束、失败回退和只读渲染边界，否则图是能偶尔出来，稳定性和可读性会一起塌。

## 共享背景

- 2026-04-14 已有规格 `读书工作台详细解读入口与思维导图路线`，确认了`详细解读`复用现有 `/report/generate` 链路，也确认了思维导图优先走 Mermaid `mindmap` 而不是直接押 NotebookLM 原生 `mind_map artifact`。
- 现在用户要补的不是新链路，而是把两个 preset 的名字和提示词重新收口。说白了，这是输出契约变了，不是再造一套功能。
- 如果只改前端按钮，不改 preset 名称、fallback 标题、提示词和错误文案，最终列表、Toast 和服务端标题会各说各话，属于那种一眼就能看出没人收过尾的改动。

## 批次顺序与依赖

- 批次01已完成现有 `/book` 行为与提示词收口，用户可直接使用新的`书籍简述`和`详细解读`。
- 批次02已完成思维导图路线决策：短期可做 Mermaid `mindmap` 渲染，长期更稳的是让模型输出结构化树，再由前端主导渲染。

## 全局约束与验收标准

- `builtin-quick-read` 仍可保留既有 ID，避免扩大接口影响半径；但对用户可见的名称、标题和提示词都要切到`书籍简述`。
- `builtin-deep-reading` 的提示词必须切换为用户提供的结构化长版说明，不再停留在旧的“逐章展开”简版。
- `/book` 页面用户可见文案、生成结果标题、相关前端测试与服务端测试必须同步更新。
- 思维导图方案输出必须明确回答 3 个问题：能不能做、主要问题是什么、有没有更好的方式。
