**工作区状态：** 未创建

**当前负责会话：** 无

## 设计结论

`快速找书` 不应继续让 `Open Library` 承担中文主检索。中文链路应改为 `豆瓣搜索 + 豆瓣详情页` 作为主书目源，继续保留现有 `ISBN -> metadata bridge` 作为微信读书入口补齐手段，并把 `微信读书公开搜索` 作为第二层补链回退；`Open Library` 只保留给英文或明确外文查询。知乎文章里那类“免费电子书下载站”不进入产品主链。

## 背景判断

- 当前服务端主链仍以 `searchOpenLibraryWithFallback()` 做候选检索，这对中文主题词天然不友好。
- 实测 `https://openlibrary.org/search.json?q=组织管理&limit=5` 仅返回 1 条拼音化旧记录，几乎不具备中文推荐价值。
- 实测 `https://book.douban.com/j/subject_suggest?q=组织管理` 能返回中文书名候选；`https://search.douban.com/book/subject_search?search_text=组织管理` 的 `window.__DATA__` 内含大量中文结果、评分、详情链接。
- 实测豆瓣详情页可直接拿到 `title`、`author`、`publisher`、`出版年`、`ISBN`、`内容简介`、`评分`、`评分人数` 等关键字段，足够支撑当前书目卡展示。
- 实测 `https://weread.qq.com/web/search/global?keyword=价值共生&count=5` 可返回结构化结果，包含 `bookId`、`title`、`author`、`publisher`、`intro`、`newRating`、`newRatingCount`；`https://weread.qq.com/web/search/books?keyword=价值共生` 的公开 HTML 结果页可直接给出 `/web/reader/...` 或 `/web/bookDetail/...` 链接。
- 但微信读书没有本项目已验证的稳定官方公开搜索 API 契约，直接把它当唯一真源，等于主动给后续维护埋雷。
- 目标是“更符合国内阅读习惯的中文找书”，不是“给用户发盗版下载入口”。这两件事混为一谈，方向就已经歪了。

## 对知乎文章的判断

- 该知乎链接正文被 `403` 拦住，未能直接读取全文；通过公开索引可确认标题是 `7个免费电子书下载网站，实现电子书自由(2025实测)`。
- 从可验证摘要看，文章核心是 `Z-Library`、`24h 搜书网` 这类免费电子书下载站或入口，不是结构化图书元数据方案。
- 这类站点的主要问题不是“技术上能不能接”，而是它们从产品方向、合规性、稳定性到可维护性都不适合做你们当前功能的主数据源。
- 结论很简单：这篇文章可以当“资源站观察样本”，不能当你们 `快速找书` 的产品方案蓝本。

## 范围边界

本次设计只定义书源策略与降级顺序：

- 明确中文主检索源应该替换 `Open Library`
- 明确各源的职责边界与失败退化
- 明确哪些候选方案应直接淘汰

本次不主张：

- 不接入盗版电子书下载站或镜像站
- 不依赖登录态、私有 RPC 或猜测式微信读书接口
- 不要求所有结果都必须有微信读书入口
- 不把微信读书推荐值伪装成通用公共评分

## 候选方案结论

### 推荐

#### 1. 豆瓣搜索 + 豆瓣详情页

- 角色：中文主检索源 + 主元数据源
- 优点：
  - 中文召回和排序明显优于 `Open Library`
  - 公开页面可直接拿到标题、作者、出版社、简介、评分、评分人数、ISBN
  - 详情页结构相对稳定，适合做受控抓取
- 缺点：
  - 没有面向第三方的正式开放 API 承诺
  - 需要维护 HTML/嵌入 JSON 解析
- 判断：最适合替代当前 `Open Library` 的中文主链

#### 2. 现有 ISBN metadata bridge

- 角色：优先补齐 `weread_url`
- 优点：
  - 你们已经接好了，不需要从零换链路
  - 与当前 `BookCandidate` 结构天然兼容
  - 可继续承担“有 ISBN 就优先桥接微信读书”的职责
- 缺点：
  - 受第三方 bridge 可用性影响
  - 无法覆盖缺 ISBN 或 bridge 漏数场景
- 判断：继续保留，作为第一层微信读书补链方案

#### 3. 微信读书公开搜索

- 角色：第二层补链回退，不做唯一真源
- 优点：
  - 中文电子书覆盖强
  - 公开搜索结果可拿到简介、推荐值、出版社和部分直接阅读链接
  - 对“国内阅读习惯”更贴近
- 缺点：
  - 接口与链接规则缺乏正式稳定承诺
  - 搜索结果会混入非标准图书内容，需要做匹配与过滤
  - 公开返回的 `newRating` 是微信读书推荐值，不应和豆瓣评分混用
- 判断：适合做微信读书入口补链，不适合做唯一书目真源

### 次选

#### 图书馆 OPAC / 联合目录

- 角色：低频 ISBN 校验与传统出版信息补全
- 优点：书目信息更偏标准化
- 缺点：没有用户向简介和评分，检索体验差，页面老旧
- 判断：可作为后续低频校验层，不进入当前主链

### 淘汰

#### 1. Open Library 继续做中文主检索

- 问题不在“调参不够”，而在中文覆盖本来就不对路。
- 让它继续扛中文主链，就是拿外文公共目录硬怼中文找书场景，属实有点自我感动。

#### 2. Z-Library / 24h 搜书网等下载站

- 它们解决的是“下书”问题，不是“稳定、结构化、合规地找书”问题。
- 没有可靠元数据契约，也不适合作为产品侧长期依赖。
- 直接淘汰，不进入实现候选。

## 推荐架构

### 1. 查询路由

- 继续复用现有意图解析结果中的 `languagePreference`
- 当查询为 `zh` 或明显中文主题词时：
  - 主链走豆瓣
- 当查询为明显英文书名、英文主题词或混合外文检索时：
  - 允许退回 `Open Library`

### 2. 豆瓣主链

- 精确书名 / 作者 / ISBN 查询：优先走 `j/subject_suggest`
- 中文主题词 / 泛主题查询：优先走 `subject_search` 页面，并从 `window.__DATA__` 里提取候选
- 对候选结果再抓取豆瓣详情页，补齐：
  - `title`
  - `authors`
  - `publisher`
  - `publishedYear`
  - `description`
  - `isbn`
  - `averageRating`
  - `ratingsCount`
  - `infoLink`
- 评分来源显式标注为 `豆瓣`

### 3. 微信读书补链

- 先保留当前 `ISBN -> metadata bridge`，这是最省改动的第一层方案
- 当 bridge 无结果、无 ISBN 或 `weread_url` 缺失时：
  - 使用 `微信读书搜索` 做高置信度补链
  - 优先按 `标题 + 作者` 匹配，而不是只按关键词瞎搜
- 若能从公开搜索结果中拿到：
  - `wereadLink`
  - `intro`
  - `publisher`
  - `newRating`
  - `newRatingCount`
  则仅作为补充字段覆盖缺失项
- 微信读书数值显示时必须标注为 `微信读书推荐值`，不得冒充公共评分

### 4. 去重与排序

- 主去重键从当前 `title + firstAuthor` 继续保留，但要优先信任豆瓣详情页返回的标准化标题/作者
- 中文主链排序优先级建议为：
  - 标题匹配度
  - 作者匹配度
  - 评分人数
  - 平均评分
  - 是否存在微信读书入口
- `Open Library` 结果仅在中文链路无可靠候选且查询偏外文时进入排序池

### 5. 失败退化

- 豆瓣搜索成功、微信读书补链失败：仍返回豆瓣候选，并明确 `微信读书：暂无数据`
- 豆瓣搜索无结果、查询含明显英文 token：退回 `Open Library`
- 豆瓣与微信读书都无可靠结果：返回当前空态，不伪造推荐
- 任何单一来源失败都不得阻断整次找书请求

## 验证方式与成功标准

- 中文主题词如 `组织管理`、`产品思维`、`领导力` 的前几条结果必须明显优于 `Open Library` 当前结果
- 返回结果默认包含中文书名、作者、出版社、简介和明确来源标签
- 有豆瓣评分时显示 `豆瓣 评分 x/10`
- 有微信读书补链时显示 `微信读书` 入口；失败时明确显示 `暂无数据`
- 实现后不得再把盗版下载站、Google Books 或纯外文目录当作中文主链来源

## 分阶段建议

### 阶段 1

- 用豆瓣替换中文主检索源
- 保留现有 metadata bridge
- `Open Library` 退到英文兜底

### 阶段 2

- 增加微信读书公开搜索补链
- 在 bridge 缺失时恢复更多微信读书入口

### 阶段 3

- 如确有需要，再增加图书馆目录作为 ISBN 校验层
- 不在这一阶段提前引入更多站点，否则维护面会迅速失控

## 自审结果

- 已明确知乎文章讨论的是免费电子书下载站，不是适合本功能的书源 API 方案
- 已明确中文主链的核心替代方案是豆瓣，不是把微信读书硬扛成唯一真源
- 已明确现有 metadata bridge 不需要立刻废弃，而是应该降级为微信读书补链层
- 已明确 `Open Library` 仍可保留为英文补充源，而不是被完全删除
- 已明确这是单任务设计文档，当前停在方案阶段，不进入实现
