## 设计结论

这轮同时收口三个问题：快速找书用户气泡必须真正按内容宽度收缩而不是整行铺满；每本书的“线上平台与评分”和“微信读书”必须拆成两行；分类标题不再固定写死为“基础教材/实践/底层逻辑”，而是由模型在受控结构中输出分类名称，但仍只允许从候选书 ID 中做归类，不允许模型创造书目。

## 背景判断

- 当前运行态中，用户气泡仍然呈现整行宽度，说明现有模板结构没有把右侧消息做成真正的内容宽度气泡。
- 当前 Markdown 每本书的“线上平台与评分”和“微信读书”挤在同一行，阅读密度差。
- 当前分区标题固定为三类预设文案，这与用户要求“让 AI 自己输出分类”冲突。
- 但分类标题动态化不能演变成让模型自由生成整本书内容，仍然只能在受控结构里返回标题和候选 ID。

## 范围边界

本次实现：

- 修正快速找书用户气泡布局，使其按内容宽度收缩并保持右侧对齐
- 调整 Markdown 模板，让“线上平台与评分”和“微信读书”拆成两行展示
- 增强各分类块之间的留白分隔
- 让模型返回动态分类标题，并按动态标题渲染分类分区

本次不实现：

- 不展示内部推理或阶段事件流
- 不让模型自由生成书目元数据
- 不把分类数量做成完全开放式无限扩展

## 推荐实现

### 1. 用户气泡

- 模板层使用左右对齐容器包裹消息气泡
- 用户气泡本体使用 `inline-block` + 明确 `max-width`
- 不依赖外层块级元素默认宽度去“碰运气”收缩

### 2. Markdown 展示

- 单本书输出改为：
  - `- 线上平台与评分：...`
  - `- 微信读书：...`
- 分类块之间增加额外空行，提升目录式阅读节奏

### 3. 动态分类标题

- 模型分类阶段输出结构从固定 `foundationIds/practiceIds/logicIds` 改为受控 section 列表
- 每个 section 仅包含：
  - `title`
  - `bookIds`
- 标题允许动态，但数量和长度要受控，避免模型输出散文化标题
- 服务端继续对白名单 ID 做校验，并在分类标题为空时回退到保守默认标题
