# 读书工作台收口A-单书模式异常处理与联调设计

**当前状态：** 已完成

## 设计结论

收口阶段不新增大功能，只解决主线接完后必然暴露的脏边界：旧 notebook 多来源残留、单书识别规则、文档间接口落差、回归验证。这个阶段不提前做，因为提前做就是在对着空气补洞。

## 依赖关系

- 依赖 `前置A-路由入口与共享壳层`
- 依赖 `前置B-PDF转文本上传接口`
- 依赖 `并行A-单书来源面板`
- 依赖 `并行B-右侧读书工具面板与快速读书`

## 目标

1. 处理单书模式与旧多来源数据的冲突
2. 完成 `/book/:id` 主流程联调
3. 回归验证 `/notebook/:id` 未被误伤

## 收口内容

### 单书识别规则

- 当 notebook 内存在多个来源时，book 页如何决定“当前书籍来源”
- 当历史来源不是 PDF 上传链路生成时，如何给出非阻塞提示
- 是否允许在 book 页直接操作旧来源，默认不允许扩散范围

### 联调校验

- 左侧上传状态与来源列表刷新是否一致
- 中间对话与报告区是否仍可用
- 右侧自动研究与快速读书是否与新页面接线一致

### 回归校验

- `/notebook/:id` 旧工作台仍可正常访问
- 旧工作台原有功能不因壳层抽离而退化

## 推荐文件边界

- `client/src/views/BookWorkbenchView.vue`
- `client/src/api/notebooks.ts`
- 必要的辅助函数与测试文件
- 若后端要补单书识别元信息，则最小修改对应接口映射文件

## 验证

1. 历史 notebook 含多来源时，book 页不会渲染崩坏
2. 单书模式提示清楚但不阻断主要流程
3. `/book/:id` 主流程手工走通
4. `/notebook/:id` 核心流程回归通过

## 自审结果

- 已把收口任务与功能任务分离，避免开发中途所有人都来抢着“顺手修一下”
