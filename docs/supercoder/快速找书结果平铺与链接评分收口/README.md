## 设计结论

`快速找书` 的结果展示应收口为单个平铺书单，不再输出 AI 分类分区，也不再单独输出“评分最高”榜单；每本书直接展示可点击链接和分平台评分信息。检索前继续保留“意图压缩”，但不引入“AI 先分类再继续检索”的分叉链路，因为这只会把当前书源流程搞乱，不会让结果更可靠。

## 背景判断

- 当前后端已经把中文主链切到 `豆瓣 + metadata bridge + 微信读书补链`，书源层已经够复杂了。
- 现有结果展示仍在做 2 到 3 个动态分类分区，再额外附一个“已核验评分最高”的榜单。说白了，就是在结果量本来不大的场景里硬凹目录感，读起来反而拧巴。
- 用户现在要的是“更直给、更能点进去”的书单，不是“模型把书重新编排成几栏”的仪式感。
- “AI 先分类再继续后续检索”听起来像高级编排，实际上会让一次查询拆成多条子检索链：每类各自召回、各自去重、各自降级、各自排序，复杂度和误判成本一起上去。为了这点虚假的结构感把架构搅乱，不值当。

## 范围边界

本次实现：

- 去掉结果中的分类分区
- 去掉“评分最高”榜单
- 结果改为单个平铺列表
- 每本书展示可点击链接（有则展示）
- 评分文案改为按平台分行展示
- 若同时存在豆瓣评分和微信读书推荐值，两者同时展示
- 前端快速找书历史识别逻辑同步兼容新的结果格式

本次不实现：

- 不引入“AI 先分类再检索”的多阶段架构
- 不新增前端结构化卡片解析器
- 不把微信读书推荐值伪装成通用评分
- 不修改书源策略本身

## 推荐实现

### 1. 服务端结果结构

- `BookFinderResult` 从“多 section + topRated”收口为单个 `results` 列表
- 列表顺序直接复用当前候选召回顺序与去重结果，不再做人为分类拆分
- 空结果时保持当前空态提示

### 2. Markdown 模板

- 顶部不恢复旧的系统标题，不额外加“快速找书结果”或“检索主题”抬头
- 每本书输出统一块结构：
  - `1. **《书名》**`
  - `- 作者：...`
  - `- 出版社：...`
  - `- 出版年份：...`
  - `- 主要内容：...`
  - `- 特色：...`
  - `- 链接：[豆瓣](...)` / `[微信读书](...)` / 多链接并列
  - `- 【豆瓣】评分：8.6/10（845 条评价）`
  - `- 【微信读书】推荐值：78.0%（664 人）`
- 若无链接，显示 `- 链接：暂无公开链接`
- 若无任何公开评分或推荐值，不显示评分行；不要用“暂无公开数据”占位污染结果块

### 3. 排序与可读性

- 不新增“榜单”区块
- 平铺列表内部保持稳定顺序即可，避免为了做展示再引入一套独立排名口径
- 评分是辅助信息，不再额外做一个“最高分”章节抢主结果焦点

### 4. 前端识别逻辑

- `createBookFinderDisplayMessages()` 不再依赖旧的分类标题或 `线上平台与评分` 文案
- assistant 消息识别改为兼容新格式，例如：
  - `- 链接：`
  - `- 【豆瓣】评分：`
  - `- 【微信读书】推荐值：`
  - 空态提示文案
- `BookFinderPanel.vue` 继续直接渲染 Markdown，不需要新增结构化 UI 容器

## 验证方式与成功标准

- 输出中不再出现分类标题和“评分最高”榜单
- 每本书有链接时，Markdown 中能渲染出可点击链接
- 豆瓣评分显示为 `【豆瓣】评分：x/10`
- 微信读书推荐值显示为 `【微信读书】推荐值：x%`
- 同一本书同时存在两种平台数值时，两行都出现
- 前端仍能正确识别并展示快速找书 assistant 历史消息

## 对“AI 先分类再检索”的判断

- 不推荐。
- 当前系统已经有：意图压缩、中文/英文路由、豆瓣主链、metadata bridge、微信读书补链、来源降级。再在检索前插一个“先分类再多路检索”，等于主动把控制流撕碎。
- 这种做法的问题不是“能不能写”，而是“写完后还值不值得维护”。对现在这个找书功能，答案很明显：不值。
- 真要让结果更聪明，后续也应该优先考虑“统一召回后的轻量重排/标签补充”，而不是检索前分叉。

## 自审结果

- 已明确本轮是单任务设计，不需要拆总览文档
- 已明确不采用“AI 先分类再检索”方案，避免架构扩散
- 已明确输出 contract 从分类分区切到平铺列表，并同步覆盖前端消息识别
