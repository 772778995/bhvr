# 工作台二期 — Studio 面板与研究质量优化

**当前状态：** 进行中

## 设计结论

将三栏布局从「标签页(来源/对话) | 报告列表/详情 | 研究控制」重构为「来源列表 | 标签页(对话/报告) | Studio 面板」。右栏从简单的研究控制升级为类似 NotebookLM 官方的 Studio 面板，支持多种 artifact 类型生成。同时优化研究报告 prompt 和问题生成策略。

### 用户交互流程

1. 用户点击某个 artifact 卡片
2. 若 SDK 支持：调用后端 API → 后端调 SDK → 返回 `artifactId` + `state: CREATING`
3. 前端弹 toast「正在生成 [类型名]…」
4. 前端定时轮询后端 artifact 状态（每 3 秒）
5. 当 state 变为 `READY`：前端获取产物数据，存入报告列表
6. 当 state 变为 `FAILED`：前端弹 toast 错误提示
7. 若 SDK 不支持（数据表）：直接弹 toast「功能正在建设中…」

### 后端 API 新增

```
POST /api/notebooks/:id/artifacts
  Body: { type: string, options?: { instructions?: string, language?: string, ... } }
  Response: { success: true, data: { artifactId: string, state: string } }

GET /api/notebooks/:id/artifacts/:artifactId
  Response: { success: true, data: { artifactId, type, state, title, ... } }

GET /api/notebooks/:id/artifacts
  Response: { success: true, data: Artifact[] }
```

### Artifact 数据与报告的关系

所有 SDK 生成的 artifact 在 NotebookLM 侧创建并存储。我们的后端不需要额外的 DB 表来存 artifact —— 直接通过 SDK 的 `artifacts.list()` 和 `artifacts.get()` 查询即可。

但为了统一在中栏「报告」tab 中展示，需要一个前端合并视图：
- 「报告」tab 同时展示：(1) 我们自己的研究报告（DB 中的 research_reports）+ (2) NotebookLM 原生 artifact 列表
- 列表项按时间倒序混合排列
- 每种 artifact 类型有不同的图标和预览方式

### 不同 Artifact 类型的预览方式

| 类型 | 预览方式 |
|------|---------|
| 研究报告（我们自己的） | Markdown 渲染，现有逻辑不变 |
| 研究报告（SDK Report） | 获取 `ReportContent`，Markdown 渲染 |
| 测验 | 获取 `QuizData`，渲染为交互式问答卡片 |
| 闪卡 | 获取 `FlashcardData`，渲染为翻转卡片 |
| 音频概述 | 内嵌 `<audio>` 播放器（base64 → blob URL） |
| 视频概述 | 显示「视频已生成」+ 在 NotebookLM 中查看的链接 |
| 幻灯片 | 显示「幻灯片已生成」+ 下载 PDF 按钮 |
| 思维导图 | 显示「思维导图已生成」提示（SDK 标记为 experimental） |
| 信息图 | 如果有 imageUrl 则 `<img>` 展示，否则提示 |

**MVP 简化**：第一版所有 artifact 的预览统一为简单的「已生成」状态卡片 + 类型图标 + 标题，点击 Report 类型展示 Markdown 内容。其他类型的富预览（音频播放、测验交互等）作为后续迭代。

## 三、来源图标修复

### 问题分析

- SDK 的 `Source` 接口**没有** icon/favicon 字段
- NotebookLM 官方 UI 前端自行基于 URL 获取 favicon
- 当前 `SourceIcon.vue` 的三级兜底逻辑（Google→DDG→/favicon.ico→globe）方向正确
- 问题在于：(1) 非 URL 类型的来源（PDF/text/drive）没有 URL，自然无 favicon (2) URL 来源的 url 字段需确保正确透传

### 修复方案

1. **非 URL 类型**：使用基于 `SourceType` 的类型图标（PDF 用文件图标、YouTube 用播放图标、Google Drive 用云盘图标、Text 用文本图标等），不再尝试获取 favicon。这些类型没有 favicon 是正常的。
2. **URL 类型**：保持现有 favicon 获取逻辑。确保 `url` 字段正确透传。
3. **可点击性**：URL 类型的来源名称/图标可点击，`window.open(url, '_blank')` 打开原网页。YouTube 类型同理，拼接 YouTube URL。非 URL 类型不可点击。

### 文件改动

- `SourceIcon.vue` — 增强 SourceType→图标映射，对非 URL 类型不再尝试 favicon 加载
- `SourcesPanel.vue` — URL 类型来源标题可点击（cursor-pointer + 外链图标）

### 当前 Prompt（routes/notebooks/index.ts:592-593）

```
请基于该笔记当前可用内容，生成一份结构化中文研究报告，包含：执行摘要、关键发现、分析过程、结论与建议。
```

### 优化后 Prompt

```
请基于该笔记本中的所有来源和此前对话内容，撰写一份系统性中文研究报告。

结构要求：
1. 执行摘要（200-300字）：概述研究背景、核心发现和关键结论
2. 研究方法与数据来源：简述所用来源类型及覆盖范围
3. 核心发现（按主题组织，每个主题需有：）
   - 主要发现陈述
   - 支撑证据与数据（引用具体来源内容）
   - 不同来源间的交叉验证或分歧
4. 深度分析
   - 因果关系与机制分析
   - 趋势与模式识别
   - 局限性与证据空白
5. 结论与建议
   - 核心结论（基于证据的确定性程度分级）
   - 具体可行的建议
   - 后续研究方向

格式要求：
- 使用 Markdown 格式
- 关键数据和引用使用引用块（>）标记来源
- 重要发现使用粗体标注
- 尽量详尽，不要省略细节
```

## 五、研究问题生成优化

### 当前问题

`chat-asker.ts` 的双会话架构中，`plannerAsker` 不知道 `visibleAsker` 实际问了什么、得到了什么回答。planner 的对话历史只有自己的 "生成下一个问题" 请求。

### 优化方案

每轮 Q&A 完成后，将问答摘要喂回 planner 会话，使 planner 了解研究进展：

```
orchestrator 循环中：
1. planner.nextQuestion() → 得到问题 Q
2. visible.askQuestion(Q) → 得到回答 A
3. 新增：planner.feedContext(`上一轮研究：问题「${Q}」→ 回答摘要「${A前200字}…」`)
4. 下一轮 planner.nextQuestion() 时，planner 知道之前问了什么、得到了什么
```

### 实现

在 `ResearchDriver` 接口新增 `feedContext(summary: string): Promise<void>` 方法。在 `createNotebookResearchDriver` 中，`feedContext` 调用 `plannerAsker` 发送一条「仅供参考，不需要回答」的上下文消息。

在 `orchestrator.ts` 的 `runAutoResearch` 循环中，每轮 Q&A 完成后调用 `driver.feedContext()`。
