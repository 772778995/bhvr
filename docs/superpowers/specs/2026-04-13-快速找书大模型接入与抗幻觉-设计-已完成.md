## 设计结论

`快速找书` 可以接入一个 OpenAI 兼容大模型，但模型只能承担“查询理解、候选分类、结果重排”这类受限职责；书名、作者、出版社、评分、阅读入口和平台可读性都必须来自可验证的书目数据源，拿不到就明确写 `未核验`，不允许模型凭空补齐。

## 背景判断

- 现有仓库已经为 `快速找书` 留了入口，但主逻辑尚未接入。
- `/book/:id` 当前只有研究历史展示，没有“点击快速找书后输入关键词并返回结果”的完整交互。
- 仓库尚未建立 `.env` 读取链路，单纯写环境变量文件不会让服务端真的用起来。

## 范围边界

本次实现只覆盖以下内容：

- 在本地 `.env` 中提供 `OPENAI_BASE_URL`、`OPENAI_TOKEN`、`OPENAI_MODEL` 三个环境变量占位
- 服务端读取上述环境变量，并通过 OpenAI 兼容接口调用外部模型
- 新增 `快速找书` 查询链路
- 前端补齐 `快速找书` 入口后的最小交互：点击入口、显示引导语、输入查询、展示返回结果
- 在软件层面对幻觉做约束，确保结果建立在外部可验证数据之上

本次不做：

- 不接入微信读书的私有或未验证抓取链路
- 不让模型直接生成书单元数据
- 不做多轮复杂会话记忆编排
- 不承诺所有场景下都能稳定凑满每个类别 5 本；若可验证结果不足，必须显式说明数量不足

## 推荐架构

### 1. 模型职责收缩

模型只做两件事：

1. 把用户自然语言查询解析成结构化检索意图
2. 在已核验的候选书目列表里做分类和排序

模型不能做：

- 直接输出书名
- 直接输出作者、出版社、评分、链接
- 直接判断“微信读书可读”并生成入口链接

### 2. 数据源职责

- 主数据源：`Google Books API`
- 补充数据源：`Open Library Search API`

服务端先拉候选，再做字段归一化、去重、排序和模板渲染。最终前端收到的是可直接展示的结构化结果和 Markdown 文本，而不是模型自由发挥后的段落。

### 3. 抗幻觉策略

必须同时满足以下约束：

1. 所有书籍必须先命中外部数据源，才能进入候选池
2. 模型分类阶段只允许返回候选书 ID，不能返回新书目
3. 服务端对模型返回的 ID 做白名单校验；不存在的 ID 一律丢弃
4. 评分、平台链接、阅读状态等字段只从外部响应里取；没有就写 `未提供` 或 `未核验`
5. 返回文案里明确标注评分来源，不冒充“微信读书评分”

### 4. 前端交互

- 左侧和右侧 `快速找书` 按钮都作为入口
- 点击后，中栏切到研究历史视图，并显示固定引导语：`您好，我是锐读，请输入您要找的书籍内容或类别`
- 中栏底部出现输入框和发送按钮
- 返回结果继续沉淀在中栏消息流内，和已有研究记录共存

### 5. 输出格式

返回结果按固定模板组织：

- 基础教材与经典类
- 行业与企业实践类
- 底层逻辑与延展类
- 已核验评分最高的 10 本书

每本书至少包含：

- 书名
- 作者
- 出版社
- 主要内容（截断到短摘要）
- 特色（截断到短摘要）
- 线上平台与评分说明

其中“线上平台与评分说明”必须诚实区分：

- 已核验：Google Books / Open Library 等公开来源
- 未核验：微信读书等当前没有可信接口的数据

- 响应中不会出现服务端未核验的虚构书籍或虚构链接
- 新增测试覆盖：配置缺失、模型返回非法 ID、模板渲染结果包含核验说明
