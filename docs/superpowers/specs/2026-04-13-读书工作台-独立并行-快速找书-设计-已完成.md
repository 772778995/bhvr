# 读书工作台独立并行-快速找书设计

**当前状态：** 已完成

## 设计结论

`快速找书` 不应直接依赖 NotebookLM 充当“全球书籍推荐引擎”，也不需要在首版接入自有 LLM。推荐首版方案是：在聊天窗口内识别“找书”意图后，后端直接调用结构化书目检索 API（优先 `Google Books API`，次选 `Open Library Search API`），再用本地规则做结果筛选、排序和固定格式输出；如后续发现用户自然语言过于发散，再补一个轻量 LLM 只负责查询改写与重排，而不是负责凭空生成书单。

## 任务定位

这是一个`独立并行`文档：

- 可以单独开 worktree 长期推进
- 不阻塞 `/book/:id` 主线页面替换
- 只依赖聊天窗口保留为主要交互入口这一前提，不依赖左/右面板先完成

## 问题判断

用户定义的交互是：

- 在聊天窗口输入类似“我要找与 xxx 相关的书籍”
- 系统输出固定格式的一组书籍数据

这件事的核心不是“对当前 notebook 来源做问答”，而是：

1. 识别找书意图
2. 提取主题词/约束
3. 查询外部书籍数据源
4. 返回结构化书单

NotebookLM 擅长的是来源内问答和带引用总结，不擅长稳定地充当结构化图书目录搜索接口。

## 方案比较

### 方案 A：直接使用 NotebookLM

可行但不推荐。

原因：

- NotebookLM 聊天回答默认受 notebook 已上传来源约束
- 它的 web research 本质是“搜网页并导入来源”，不是干净的书目检索接口
- 为了找几本书去污染 notebook 来源，非常蠢，后续研究上下文会被搜索噪音塞满
- 输出格式稳定性差，难以保证每次都吐出你想要的固定字段
- 成本、时延和结果可控性都不适合作为聊天里的轻交互能力

结论：NotebookLM 可以做“辅助理解当前书籍主题”，但不该做“书目检索主引擎”。

### 方案 B：接入自有 LLM

首版不推荐直接上。

原因：

- 你现在要的核心能力是找书，不是开放式生成
- 真正决定结果质量的是书目数据源，而不是模型嘴有多会说
- 如果没有结构化书目库，LLM 只会一本正经地胡说八道，推荐出不存在或信息残缺的书，这不是功能，是事故
- 当前项目基线也明确偏向“不引入外部 LLM”

结论：只有当用户输入足够自然、模糊、需要复杂语义改写时，才考虑把 LLM 作为“查询改写器/重排器”补进来。

### 方案 C：结构化书目 API + 本地规则

这是推荐方案。

推荐组合：

- 主数据源：`Google Books API`
- 次数据源：`Open Library Search API`

理由：

- `Google Books API` 提供公开 volume 搜索、作者、出版日期、分类、简介、封面、预览/购买链接等结构化字段，适合产出稳定卡片
- `Open Library` 适合作为免费补充源，能补 cover、作者、edition、可读性等信息
- 这类 API 的返回是结构化元数据，不需要模型先编再救火
- 输出固定格式完全可以由后端模板化生成，稳定、便宜、可测试

## 推荐架构

### 首版

1. 用户在聊天窗口输入“我要找与 xxx 相关的书籍”
2. 前端或后端识别为 `快速找书` 意图
3. 后端提取搜索词与可选约束：
   - 主题
   - 作者
   - 语言
   - 是否偏入门/进阶
4. 后端查询 `Google Books API`
5. 若结果不足或字段不完整，再补查 `Open Library`
6. 本地做筛选和重排
7. 将结果按固定 schema 返回到聊天窗口

固定 schema 建议：

- `title`
- `authors`
- `publishedYear`
- `publisher`
- `categories`
- `description`
- `coverUrl`
- `previewLink`
- `infoLink`
- `whyRecommended`

其中 `whyRecommended` 首版可不用 LLM，直接用模板生成，例如：

- “匹配主题词：创业、组织行为”
- “出版信息完整，且有预览链接”
- “分类与查询主题高度相关”

### 二期增强

若后续用户输入开始明显超出规则提取能力，例如：

- “给我找偏组织心理学、不要太鸡汤、最好近十年的中文书”

再引入一个小 LLM 做两件事：

1. 查询改写
2. 检索结果重排

注意：

- LLM 只负责理解和排序
- 书籍元数据仍然必须来自外部书目 API
- 不允许让 LLM 直接凭空输出书名、作者和出版信息

## NotebookLM 在这个功能里的合理位置

NotebookLM 不是主引擎，但可以作为可选辅助：

- 如果需求是“找与当前上传这本书相关的延伸阅读”
- 可以先让 NotebookLM 从当前书里提取主题、关键词、作者流派、研究对象
- 再把这些关键词交给书目 API 去查

这样它做的是“从当前书生成检索线索”，不是“自己瞎搜书”。这个姿势才没那么抽象失控。

## 现成方案探索结论

### 推荐

1. `Google Books API`
   - 官方公开 API，支持 volume 搜索与丰富元数据
   - 适合做首版主数据源
2. `Open Library Search API`
   - 免费、开放、覆盖面广
   - 适合作为补充与兜底数据源

### 次选

1. NotebookLM web research
   - 能搜网页，但不是书目检索接口
   - 适合研究，不适合轻量找书

### 淘汰

1. 让 NotebookLM 直接在聊天里生成书单
   - 结果不可控，格式不稳，且与 notebook 来源边界冲突
2. 首版直接上自有 LLM 生成书单
   - 成本高、幻觉风险高、收益不成比例

## 实施建议

首版实现顺序：

1. 先把 `快速找书` 做成聊天中的显式意图分支或命令
2. 接 `Google Books API` 出稳定结构化结果
3. 固定前端展示卡片格式
4. 若结果覆盖不足，再补 `Open Library`
5. 最后再评估是否需要 LLM 做查询改写

## 回答当前问题

结论很直接：

- **不建议直接靠 NotebookLM 做这个功能**
- **首版也不需要先接自己的 LLM**
- **最合理的是“书目 API 检索 + 本地固定格式输出”，必要时再补一个轻量 LLM 做查询理解**
