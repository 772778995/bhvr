## 设计结论

这轮只做展示收口，不把数据源切成“只用微信读书”。原因很直接：当前项目没有稳定、结构化、已验证的微信读书公开 API，强上只会把抗幻觉方案推翻。可立即落地的部分包括：去掉面板顶部说明文案、修复长链接溢出、增强每本书之间的分隔表现。

## 背景判断

- `BookFinderPanel` 顶部仍有一段说明性文案，用户已明确要求去掉
- Markdown 渲染区域对超长链接缺少强制换行约束
- 当前书目条目之间只有列表序号，视觉分隔偏弱
- 当前输出里使用了 Google Books / Open Library 链接，这与“只要微信读书链接”诉求冲突
- 但微信读书目前仅能确认存在公开网页内容，无法确认可稳定依赖的公开结构化接口

## 范围边界

本次实现：

- 去掉快速找书页顶部说明文案
- 修复超长链接导致的气泡溢出
- 强化每本书之间的视觉分隔
- 保持当前公开数据源声明，不伪造微信读书评分和入口

本次不实现：

- 不把结果切换为仅微信读书数据源
- 不抓取未验证的微信读书私有接口
- 不伪造“微信读书评分”和“微信入口链接”字段

## 推荐方案

### 1. 顶部文案

- 删除 `BookFinderPanel` 顶部那句“只展示快速找书自己的欢迎语……”
- 欢迎信息仍只保留首条 assistant 气泡

### 2. 长链接换行

- 在 `.prose-warm a` 与相关容器上增加 `overflow-wrap: anywhere` / `word-break: break-word`
- 让长链接在气泡内自然折行，不再把版心撑爆

### 3. 书目分隔

- 保持 Markdown 输出，但在每本书之间插入 `---`
- 这样不用额外引入复杂卡片语法，也能和现有 Markdown 渲染链路兼容

### 4. 微信读书边界

- 结果字段里继续诚实写 `微信读书：未核验`
- 如未来需要真接微信读书，必须先补一轮独立的“数据源可行性与合规性”规格，不在这轮偷渡

- 已明确这轮能做的是展示收口，不能假装完成微信读书数据接入
- 已明确分隔线方案比新增卡片语法更小、更稳
- 已明确微信读书诉求需要独立数据源方案，不在这轮偷做
