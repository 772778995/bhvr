# NotebookLM 工作台壳层设计

**当前状态：** 已完成

## 设计结论

先搭工作台壳和接口壳，读写分层，未接能力统一占位。

## 1. 背景

当前项目已经具备一套围绕自动课题研究的后端流程和基础前端页面，但还没有一个接近 Google NotebookLM 工作台形态的主界面。

本次设计的目标不是完整复刻 NotebookLM，也不是立即打通全部真实能力，而是先搭建一个可演进的工作台壳子：

- 前端先形成接近 NotebookLM 的单页工作台结构
- 后端先形成按能力域分组的接口壳
- 页面能正常展示和交互
- 未接入能力统一返回“功能正在建设中”
- 后续逐步替换为真实接口和业务能力

用户已明确要求：

- 优先做单页 `notebook/:id` 详情工作台
- 整体结构参考 Google NotebookLM，但不要求像素级还原
- 顶部只保留简化版标题和少量按钮
- 读取类接口返回假数据，写入类接口返回“功能正在建设中”
- 自动课题研究能力本次只保留入口占位，不急于接入
- 设计文档不写具体代码实现，必要时仅写伪代码

## 2. 目标与非目标

### 2.1 目标

- 新增一个接近 NotebookLM 工作台的信息架构和页面骨架
- 明确前后端的分层边界，避免后续边做边改接口
- 让前端页面可以在假数据支持下完整渲染
- 为未来接入 `notebooklm-kit` 实际能力和已有自动课题研究能力预留稳定入口

### 2.2 非目标

- 不做 Google NotebookLM 的 1:1 视觉复刻
- 不在本次设计中实现真实来源上传、真实对话、真实 Studio 生成能力
- 不在本次设计中接入自动课题研究的真实执行链路
- 不扩大范围去重构现有 research 模块

## 4. 总体方案

采用“页面先像、接口先分层建壳”的方案。

### 4.1 前端职责

前端负责：

- 提供 `notebook/:id` 单页工作台
- 依据三栏结构渲染 notebook 数据
- 对未实现能力统一做友好提示
- 将旧的自动研究页面保留为独立模块，不直接混入新工作台

### 4.2 后端职责

后端负责：

- 新增 NotebookLM 风格接口分组
- 为读取类接口提供稳定的假数据响应
- 为写入类接口提供统一的“建设中”响应
- 为后续真实转发 `notebooklm-kit` 和自动研究能力预留路由边界

### 4.3 分阶段演进原则

第一阶段：

- 页面壳跑通
- 接口壳跑通
- 数据使用 stub

第二阶段：

- 将来源、对话、Studio 的部分读取能力逐步替换为真实数据

第三阶段：

- 将自动课题研究入口与已有研究系统衔接

## 5. 页面设计

### 5.1 路由范围

新增核心页面：

- `GET /notebook/:id` 对应前端路由 `notebook/:id`

本次不要求新增完整列表页、设置页或多级嵌套路由。第一版先围绕单个 notebook 工作台展开。

### 5.2 页面结构

页面由四个主要区域组成：

1. 顶部简化工具栏
2. 左侧来源栏
3. 中间对话栏
4. 右侧 Studio 栏

伪结构如下：

```text
WorkbenchPage
├─ TopBar
│  ├─ NotebookTitle
│  └─ PrimaryActions(1~2 items)
├─ SourcesPanel
├─ ChatPanel
└─ StudioPanel
```

### 5.3 顶部简化工具栏

顶部区域采用“保留气质，不追求完整复刻”的策略。

保留内容：

- notebook 标题
- 1 到 2 个操作按钮

暂不纳入第一版：

- 完整分享能力
- 完整设置页
- 用户头像菜单
- 所有二级菜单与弹层

目标是让页面视觉上具备“这是一个工作台”的入口感，而不是把时间投入在低价值按钮上。

### 5.4 来源栏

来源栏承担 notebook 材料管理入口的角色。

第一版建议包含：

- 标题：`来源`
- 添加来源按钮
- 搜索来源输入区域
- 一组来源筛选/提示入口占位
- 来源列表区域
- 空状态说明

交互原则：

- 有来源时展示列表卡片
- 无来源时展示说明文案
- 点击添加来源后，如果能力未接入，则返回统一建设中提示

### 5.5 对话栏

对话栏承担 notebook 交互主工作区角色，应作为视觉焦点。

第一版建议包含：

- 标题：`对话`
- 中部欢迎空状态
- 底部输入框区域
- 发送按钮
- 消息列表区域

交互原则：

- 当没有消息时，展示类似截图中的“先添加来源再开始使用”空状态
- 输入区始终存在，保持页面像“可操作产品”
- 点击发送时，如果对话真实能力未接入，则显示建设中提示

### 5.6 Studio 栏

Studio 栏承担能力扩展区角色，也是未来接自动研究入口的最佳位置。

第一版建议包含：

- 标题：`Studio`
- 一组工具卡片
- 下方输出空状态说明
- 底部主入口按钮

推荐工具卡片先按截图风格保留以下名称：

- 音频概览
- 演示文稿
- 视频概览
- 思维导图
- 报告
- 闪卡
- 测验
- 信息图
- 数据表格

另外增加一个独立卡片或按钮：

- 自动课题研究

该入口第一版只做占位，点击后统一提示“功能正在建设中”。

### 5.7 视觉还原边界

第一版追求以下相似性：

- 三栏分区清晰
- 卡片式面板
- 圆角与浅边框
- 大面积留白
- 中间工作区更突出
- 空状态居中呈现

第一版不追求：

- 像素级一致的图标与按钮样式
- 所有 hover、popover、menu 动画
- 完整响应式复刻 Google 原站每一个细节

## 6. 后端接口设计

### 6.1 设计原则

后端接口以“前端页面分区一一对应”为核心原则。每一个页面区域都应有独立、清晰的接口归属，避免未来业务接入时出现接口职责不清。

建议按以下能力域分组：

- `notebooks`
- `sources`
- `chat`
- `studio`
- `research`

### 6.2 接口分组建议

#### notebooks

- `GET /api/notebooks/:id`
  - 返回 notebook 基础信息

#### sources

- `GET /api/notebooks/:id/sources`
  - 返回来源列表
- `POST /api/notebooks/:id/sources`
  - 当前返回“功能正在建设中”

#### chat

- `GET /api/notebooks/:id/chat/messages`
  - 返回消息列表
- `POST /api/notebooks/:id/chat/messages`
  - 当前返回“功能正在建设中”

#### studio

- `GET /api/notebooks/:id/studio/tools`
  - 返回工具卡片列表与可用状态
- `POST /api/notebooks/:id/studio/:tool`
  - 当前返回“功能正在建设中”

#### research

- `GET /api/notebooks/:id/research`
  - 返回自动课题研究入口状态
- `POST /api/notebooks/:id/research`
  - 当前返回“功能正在建设中”

### 6.3 真实能力与 stub 的边界

第一版接口策略明确为：

- 读取类接口：返回稳定 stub 数据
- 写入类接口：统一返回未实现提示

未来逐步接入时，只替换接口内部实现，不修改前端页面结构和接口路径。

### 6.4 notebooklm-kit 对接原则

对 `notebooklm-kit`，本次只做“未来对接边界设计”，不要求完整实现。

设计原则：

- 已知可用能力未来优先映射到相同能力域
- 当前尚不明确或尚不准备接入的接口保留 stub
- 不为了“接口看起来全”而在第一版设计中引入过多未经验证的细分能力

## 7. 数据模型设计

第一版只保留支撑页面渲染所需的最小数据模型。

### 7.1 Notebook

字段建议：

- `id`
- `title`
- `description`
- `updatedAt`

### 7.2 Source

字段建议：

- `id`
- `title`
- `type`
- `status`
- `summary`

### 7.3 ChatMessage

字段建议：

- `id`
- `role`
- `content`
- `createdAt`
- `status`

### 7.4 StudioTool

字段建议：

- `id`
- `name`
- `description`
- `available`

### 7.5 ResearchEntry

字段建议：

- `id`
- `name`
- `status`
- `message`

### 7.6 统一响应结构

建议统一使用简单响应格式：

- `success`
- `data`
- `message`
- `errorCode`

统一响应格式的价值：

- 前端更容易处理 stub 和真实响应
- 未实现与真实失败可以用一致方式展示
- 后续替换接口内部实现时，不需要同步改动大量前端逻辑

## 8. 页面数据流

### 8.1 初始加载

进入 `notebook/:id` 后，页面应并行请求：

- notebook 基础信息
- 来源列表
- 对话消息列表
- Studio 工具列表
- 自动课题研究入口状态

### 8.2 页面渲染逻辑

三个栏位分别根据返回数据独立渲染：

- 来源栏显示来源列表或空状态
- 对话栏显示消息列表或欢迎空状态
- Studio 栏显示工具卡片和空输出说明

### 8.3 交互反馈逻辑

当用户点击未实现能力时，前端统一执行：

```text
if action is not implemented:
  show friendly message("功能正在建设中")
  do not crash
  do not navigate to blank page
```

该策略适用于：

- 添加来源
- 发送消息
- 点击 Studio 工具卡片
- 点击自动课题研究入口

## 9. 错误处理与占位策略

### 9.1 页面级空状态

当没有来源、没有消息、没有 Studio 输出时，页面应提供清晰空状态说明，而不是空白面板。

### 9.2 操作级建设中提示

所有尚未接入的操作必须给出一致提示：

- 文案统一为：`功能正在建设中`

### 9.3 接口级未实现响应

未实现写入类接口应统一返回结构化未实现结果。前端根据返回内容展示友好提示，不直接暴露技术错误。

### 9.4 非法 notebook id

当路由中的 notebook id 缺失、非法或读取失败时：

- 页面应展示明确错误态
- 不允许出现白屏
- 不允许仅在控制台打印报错而缺少界面反馈

## 10. 与现有自动研究模块的关系

现有自动研究模块本次保持独立，不直接并入新工作台主流程。

本次只在 Studio 区或相关入口区域中预留一个“自动课题研究”入口卡片或按钮。该入口用于说明未来将与现有能力整合，但在第一版不触发真实研究任务。

这样处理的好处是：

- 避免本次改动牵连现有稳定流程
- 先建立产品层面的统一工作台形态
- 后续接入自动研究时有明确落点

## 11. 验收标准

第一版完成后，应满足以下标准：

- 存在新的 `notebook/:id` 页面
- 页面整体结构接近 Google NotebookLM 的三栏工作台
- 顶部有简化版标题区和少量按钮
- 左中右三栏都有稳定占位内容与空状态
- 页面请求新的 Notebook 风格接口，而不是复用旧 research 页面接口
- 读取类接口能让页面完整渲染
- 写入类接口统一返回“功能正在建设中”
- Studio 中包含自动课题研究入口卡片或按钮
- 现有 research 页面和已有流程不被破坏

## 12. 伪代码级实施轮廓

本节仅用于说明职责边界，不代表具体实现代码。

```text
Frontend:
  open /notebook/:id
  load notebook + sources + messages + studio tools + research entry in parallel
  render three-panel workbench
  for unfinished actions:
    show "功能正在建设中"

Backend:
  expose notebook-oriented route groups
  return stub data for read endpoints
  return structured not-implemented response for write endpoints
  keep real implementation behind the same route boundaries for later replacement
```

## 13. 后续规划建议

本设计确认后，下一步应进入实现计划编写，而不是直接编码。

建议的计划拆分顺序：

1. 前端工作台路由与三栏页面骨架
2. 后端 Notebook 风格路由壳
3. 统一 stub 数据与统一未实现响应
4. Studio 自动课题研究入口占位
5. 验证旧 research 模块未受影响

该顺序有利于先看到页面成果，再逐步填充接口边界。
